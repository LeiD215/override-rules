## 案例 #1: MyDirectCDN provider 分支名写错

**时间**: 2026-07-21  
**影响范围**: MyDirectCDN 规则提供者无法加载  
**严重程度**: 中（功能失效，但不影响其他规则）

### 症状

- Sub-Store 生成的配置中，`MyDirectCDN` 规则提供者面板显示为空
- 客户端无法拉取到 `MyDirectCDN` 的规则内容
- 但其他自定义规则（如 AI 服务分组）正常工作

### 原因

在 `src/rule_providers.ts` 中，`MyDirectCDN` 的 URL 配置错误：

```typescript
// 错误配置
url: 'https://cdn.jsdelivr.net/gh/LeiD215/override-rules@master/ruleset/MyDirectCDN.list'

// 正确配置
url: 'https://cdn.jsdelivr.net/gh/LeiD215/override-rules@main/ruleset/MyDirectCDN.list'
```

**根因**: 
- 仓库的默认分支是 `main`，不是 `master`
- 在配置 URL 时，习惯性使用了 `@master`，没有核实仓库的实际分支名
- 这是一个"抄现有代码"导致的错误：其他第三方规则提供者（如 TikTok、EHentai）可能确实使用 `master` 分支，但本仓库使用 `main`

### 解决过程

1. **发现问题**: 用户在 Sub-Store 中查看规则提供者面板，发现 `MyDirectCDN` 为空
2. **排查**: 
   - 检查 `src/rule_providers.ts` 中的 URL 配置
   - 发现 URL 中使用了 `@master`，但仓库实际分支是 `@main`
3. **修复**: 将 URL 中的 `@master` 改为 `@main`
4. **验证**: 
   - 本地构建通过
   - 发布新版本 `src-v2.5.7`
   - 用户在 Sub-Store 中刷新后，`MyDirectCDN` 正常加载

### 教训

1. **不要盲目复制**: 即使是"看起来一样"的配置，也要核实具体值是否正确
2. **URL 配置要谨慎**: 分支名、路径等细节错误会导致资源无法加载，且不容易被发现
3. **验证环节要完整**: 本地构建通过不代表配置正确，需要在实际环境中验证资源是否可访问

### 影响

- **人工介入时长**: 约 30 分钟
- **是否需要回滚**: 否（直接修复，不需要回滚）
- **是否影响外部用户**: 是（Sub-Store 生成的配置里 MyDirectCDN 规则提供者面板显示不出内容，客户端无法拉取规则）

## 案例 #2: 低倍率节点分组移除后残留引用

**时间**: 2026-07-22  
**影响范围**: iOS 客户端（Clash Mi）启动失败  
**严重程度**: 高（客户端完全无法使用）

### 症状

- iOS 客户端（Clash Mi）启动时报错：
  ```
  proxy group[6]: 选择代理: '低倍率节点' not found
  ```
- 客户端完全无法启动，用户无法使用任何代理功能

### 原因

在 2026-07-21 的"移除低倍率节点分组"操作中，只删除了 `src/proxy_groups.ts` 中的分组定义，但遗漏了 `src/selectors.ts` 中的引用：

```typescript
// src/selectors.ts 中的残留代码
const lowCost = lowCostNodes.length > 0 || regexFilter;

// 在构建 defaultSelector / defaultProxies / defaultProxiesDirect 时
buildList(..., lowCost && PROXY_GROUPS.LOW_COST, ...)
```

**根因**:
- 删除操作不彻底，只关注了"主要定义"，没有全文搜索确认没有残留引用
- `lowCostNodes` 在多个文件中被引用（`constants.ts`、`node_parser.ts`、`main.ts`、`types.ts`、`selectors.ts`），形成了一条完整的死代码链
- 类型检查（`tsc --noEmit`）无法捕获字符串字面量引用（如 `'低倍率节点'`）

### 解决过程

1. **发现问题**: 用户在 iOS 客户端启动时看到报错
2. **排查**: 
   - 检查 `src/proxy_groups.ts`，确认"低倍率节点"分组已删除
   - 全文搜索 `低倍率节点`，发现 `src/selectors.ts` 中仍有引用
   - 进一步搜索 `lowCost` / `LOW_COST`，发现多个文件中都有残留
3. **修复**: 
   - **阶段 1**: 修复 `src/selectors.ts`，删除 `lowCostNodes` 参数和相关逻辑
   - **阶段 2**: 全量清理死代码链
     - `src/constants.ts`: 删除 `LOW_COST_NODE_MATCHER` 常量
     - `src/node_parser.ts`: 删除 `parseLowCost` 函数
     - `src/main.ts`: 删除 `lowCostNodes` 相关逻辑
     - `src/types.ts`: 删除 `lowCostNodes` 类型定义
     - `src/selectors.ts`: 删除 JSDoc 中的 `@param input.lowCostNodes`
4. **验证**: 
   - `npx tsc --noEmit` 通过
   - `npm run build` 通过
   - `npm run lint` 通过
   - 功能验证：14 个真实节点数据调用 `main(config)`，确认"低倍率节点"分组不存在，所有候选列表无残留引用
   - 全文搜索确认：`src/` 下已无 `lowCost` / `LowCost` / `LOW_COST` / `parseLowCost` 任何残留
5. **发布**: 发布新版本 `v2.5.9`

### 教训

1. **删除操作要彻底**: 删除一个功能时，不能只删"主要定义"，必须全文搜索确认没有残留引用
2. **类型检查不是万能的**: `tsc --noEmit` 能捕获部分残留引用，但不能捕获所有（比如字符串字面量引用）
3. **功能验证是最后一道防线**: 用真实节点数据调用 `main(config)` 是发现这类问题的关键步骤
4. **分阶段修复**: 对于复杂的死代码链，可以分阶段修复（先修复关键路径，再全量清理），但每阶段都要完整验证

### 影响

- **人工介入时长**: 约 2 小时（阶段 1 + 阶段 2）
- **是否需要回滚**: 否（直接修复，不需要回滚）
- **是否影响外部用户**: 是（iOS 客户端完全无法启动）
