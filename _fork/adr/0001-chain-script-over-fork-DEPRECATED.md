# ADR-0001: 用"后处理脚本"叠加自定义规则，而不是 Fork 上游仓库

- **状态**: superseded（被 ADR-0003 取代）
- **日期**: 2026-07-21
- **决策者**: Hermes Agent

## 背景 Context

上游 `powerfullz/override-rules` 项目提供了一套完整的规则覆写方案，但用户希望在此基础上叠加自己的自定义规则（AI 服务分组、静态资源直连等）。需要决定：是 Fork 上游仓库直接改源码，还是写一个独立的"后处理脚本"在上游生成的配置基础上做二次加工？

## 考虑过的选项 Options Considered

### 选项 1: Fork 上游仓库，直接改源码
- **优点**: 
  - 改动直接，不需要额外的脚本层
  - 可以复用上游的构建流程（`npm run build`）
- **缺点**: 
  - 上游更新时需要手动 merge，可能产生冲突
  - 需要维护一个长期 diverge 的 fork

### 选项 2: 写独立的"后处理脚本"（Chain Script）（选定）
- **优点**: 
  - 上游更新时不需要 merge，直接拉取最新配置再加工
  - 自定义逻辑完全独立，不影响上游代码
  - 可以灵活组合多个处理步骤
- **缺点**: 
  - 需要额外维护一个脚本
  - 需要自己处理配置的加载和输出

## 决定 Decision

选择**选项 2**：写独立的"后处理脚本"（Chain Script）。

具体实现：
- 创建 `scripts/custom-override.js`，读取上游生成的配置，做二次加工
- 通过 GitHub Actions 自动执行：先拉取上游最新配置，再运行自定义脚本，最后输出最终配置

## 理由 Rationale

1. **降低维护成本**: 上游更新时不需要处理 merge 冲突，只需要确保自定义脚本能兼容新的配置格式
2. **灵活性**: 可以独立迭代自定义逻辑，不受上游发布节奏限制
3. **可追溯性**: 自定义逻辑集中在一个脚本里，便于审查和调试

## 不确定性 Uncertainties

（迁移时补充，非原决定撰写时内容）

- 后处理脚本的性能开销是否可接受

## 影响 Consequences

- 创建了 `scripts/custom-override.js` 作为自定义逻辑的入口
- 配置了 GitHub Actions 工作流，自动执行"拉取上游 → 自定义加工 → 输出最终配置"的流程
- 后续发现这个方案有问题（见 ADR-0003），最终改为直接 Fork 上游仓库
