# ADR-0001：用"后处理脚本"叠加自定义规则，而不是 Fork 上游仓库

- **状态**：❌ 已废弃，被 ADR-0003 取代
- **日期**：2026-07-21

## 背景 Context
上游 [powerfullz/override-rules](https://github.com/powerfullz/override-rules) 会持续更新
`convert_min.js`（Sub-Store 订阅转换脚本，用于把节点列表生成完整的 mihomo 分流配置）。
我需要在它的基础上加自己的自定义规则（域名分流、策略组、DNS/广告拦截），并且要求
上游每次更新后，自定义规则依然能"套用"进去，长期可持续维护。

## 考虑过的选项 Options Considered
1. **Fork 仓库 + Git patch/rebase**：把上游 clone 下来改，每次上游更新后 `git rebase` /
   `cherry-pick` 自己的改动。
   - 优点：改动直接、直观
   - 缺点：上游是压缩后的 `convert_min.js`（源码在 `src/*.ts`），语法/结构变化时
     patch 很容易冲突；每次都要手动解决冲突，长期维护成本高，且冲突解决过程本身
     不产生自然的"记录"
2. **链式后处理脚本（Chain Script）**：上游脚本原样不动，另写一个脚本，在上游生成的
   完整配置对象之上做增删改，两者顺序执行
   - 优点：与上游完全解耦，上游内部实现怎么变都不影响我；自定义逻辑集中在一个文件里，
     天然适合版本控制和记录
   - 缺点：依赖上游输出的配置对象结构（`proxy-groups` / `rules` / `dns` 等字段名）保持
     稳定；如果上游做了破坏性重构（字段改名等），后处理脚本需要跟着调整——但这种情况
     远少于源码级冲突，而且容易通过跑一次测试就能发现

## 决定 Decision
采用**链式后处理脚本**方案：上游脚本不动，自定义逻辑全部写在
`scripts/custom-override.js` 里，对上游输出的配置对象做二次加工。

## 理由 Rationale
- 冲突面从"整个源码"缩小到"配置对象的字段结构"，大幅降低维护成本
- 自定义逻辑和上游逻辑物理分离，天然满足"完整流程可追溯"的要求——每次改动只发生
  在一个文件里，git log 就是完整历史
- 后续无论上游怎么更新，"套用自定义规则"这一步基本不需要人工介入

## 影响 Consequences
- 好处：长期维护成本低，历史记录天然清晰
- 代价：需要有一个"跑一遍看看有没有报错/字段缺失"的测试步骤（见 RUNBOOK），
  不能完全无脑自动合并——上游若做破坏性重构，脚本会需要跟着更新（此时应新开一条
  ADR 记录相应调整）

## 补充决定（2026-07-21）：执行机制选 GitHub Actions 后处理，不用客户端链式覆写

**背景**：调研确认，"链式"这件事确实可行，但支持它的不是 Sub-Store 本身，而是实际
使用的客户端（Mihomo Party / Clash Verge 的"覆写"功能）——客户端按你排列的顺序，
依次对 `config` 执行多个覆写脚本。这意味着链式执行依赖你用的具体客户端是否支持、
以及客户端本身的更新不出岔子。

**考虑过的选项**：
1. **客户端原生链式覆写**：把 `convert_min.js` 和 `custom-override.js` 都配置成
   客户端的覆写条目，按顺序排列——优点是不需要额外基础设施；缺点是绑定具体客户端
   实现，换客户端/客户端更新覆写机制都可能受影响，而且这个过程不会自然产生任何
   执行记录
2. **GitHub Actions 后处理（已选定）**：CI 定时拉取上游生成的基础配置，跑
   `custom-override.js` 处理，产出最终订阅文件并提交——优点是不依赖任何特定客户端；
   每次运行都是一次 CI 日志，天然留痕，和"完整流程记录"的原则直接对上；换客户端、
   换设备完全不受影响

**决定**：采用 GitHub Actions 后处理方案。

**理由**：
- 不想被绑定在某个具体客户端的覆写实现上（对应用户顾虑 1）
- 想要更完整的自动化执行记录——CI 运行本身就是留痕，不需要额外维护（对应用户顾虑 2）
- 更看重长期稳定性，不希望客户端更新影响到这套机制（对应用户顾虑 3）
- **额外考虑**：如果上游 `powerfullz/override-rules` 未来停止维护，GitHub Actions
  方案下，"生成基础配置"这一步是可以自己接管、替换掉上游脚本的（毕竟只是一个
  CI 步骤），而客户端覆写链式的方式，一旦上游脚本停更，整套东西更难独立维持

**影响**：
- 需要额外维护一套 CI 配置（`.github/workflows/sync-and-build.yml`），比纯客户端
  覆写多一层基础设施
- 后续所有关于"怎么拉取上游 / 怎么跑后处理 / 怎么产出最终订阅"的实现细节，应该
  在 RUNBOOK 里补全（当前 `sync-and-build.yml.example` 只是骨架，`build.js` 的
  具体实现还没写）
