# ADR-0002：用自维护小名单替代上游的静态资源/CDN 规则集

- **状态**：已采纳（决策内容不变；实现位置随 [ADR-0003](0003-fork-upstream-typescript-source.md) 迁移，见下方"实现位置"更新说明）
- **日期**：2026-07-21

## 背景 Context
上游生成的配置里，`StaticResources`/`CDNResources`/`AdditionalCDNResources` 这三个
规则集，指向 skk.moe / jsdelivr 上维护的大型域名列表（其中 `StaticResources` 单个
文件就有 4500+ 条，每天自动更新），本意是给"按流量计费的机场"用户提供"通用 CDN /
静态资源单独直连、节省代理流量"的选项。

排查静态资源"抢跑"问题时，发现这份列表混进了大量本该属于具体服务（AI服务/
Netflix/TikTok/Twitter/Github 等）的域名，做了一次"提前顺序"的补丁修复。但这次
修复只解决了"当前已知的撞车"，列表本身仍然是一个不受控、每天都在变化的第三方
黑盒——以后任何时候它新增域名、恰好撞上某个服务，都可能重新引入同样的问题，属于
持续性风险，不是一次性修完就没事。

同时，这份规则集存在的原始动机（省流量）对使用自建 VPS（不按流量计费）的场景基本
不成立。

## 考虑过的选项 Options Considered
1. **维持现状**：继续用上游大列表，靠"提前名单"的补丁持续应对撞车问题
   - 优点：改动最小
   - 缺点：问题根源没解决，列表每天更新，坑是第三方持续在挖，需要长期盯着
2. **直接删除**：把这三个规则集从配置里整体去掉
   - 优点：彻底消除风险，配置最简单
   - 缺点：失去"部分明确想直连的 CDN 域名"这个能力，这些域名会退化到
     GEOIP,cn / GFWList / MATCH,Final 的兜底逻辑里
3. **自维护小名单（已选定）**：去掉上游大列表，换成一份自己维护、体积很小
   （几十条量级）、只包含明确知道且想直连的域名的清单
   - 优点：兼顾"部分场景确实想直连"的价值，同时把维护成本和风险都降到可控范围
     （不受第三方每天更新影响，出问题也只可能是自己加错，排查范围小）
   - 缺点：需要自己维护这份清单，覆盖面不如上游大列表广

## 决定 Decision
采用自维护小名单方案。移除对 `StaticResources`/`CDNResources`/
`AdditionalCDNResources` 三个 rule-provider 及其对应规则的引用，改为维护一份
自己的直连域名清单，插入到原来"静态资源"规则所在的位置之后（具体位置见
ADR-0003 里的补充：挪到了所有专属服务规则之后，进一步降低撞车风险）。

## 理由 Rationale
- 省流量这个原始动机不成立，没必要为了一个用不上的能力承担持续性风险
- "有些域名我确实清楚、想直连"是真实需求，不该因式 2（直接删除）而牺牲掉
- 几十条自维护清单 vs 几千条第三方每天更新的清单，前者的可控性和可维护性都
  明显更好，出问题时排查范围也小得多

## 影响 Consequences
- 好处：彻底消除了"静态资源抢跑其他服务"这类问题以后重复出现的可能性；配置
  更简单，不再依赖三个远程规则集的更新
- 代价：需要自己整理这份直连域名清单，覆盖面比上游列表窄——但这本来就是
  权衡后主动接受的取舍，不是遗漏

## 实现位置（随 ADR-0003 更新）
- Rule Provider 定义：`src/rule_providers.ts` 里的 `MyDirectCDN`
- 清单内容：`ruleset/MyDirectCDN.list`（classical 文本格式，一行一条
  `DOMAIN-SUFFIX,域名`，支持 `#` 注释）
- 规则归属：`src/rules.ts` 里的 `RULE-SET,MyDirectCDN,DIRECT`，位置在所有
  专属服务规则之后
- 起点内容：5 个国内 CDN 镜像域名（staticfile.org / staticfile.net /
  bootcdn.net / bootcss.com / baomitu.com）
