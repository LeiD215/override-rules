# ADR-0003：改用官方推荐的 Fork + 源码定制方案，取代 ADR-0001 的独立后处理项目方案

- **状态**：已采纳（**取代 ADR-0001**）
- **日期**：2026-07-21

## 背景 Context
ADR-0001 定下的方向是：不 fork 上游，自己另建一个独立项目，写一个后处理脚本
（`custom-override.js`）在上游生成的配置基础上做二次加工，通过 GitHub Actions
拉取 → 加工 → 产出最终配置文件。

这个方向从根上就理解错了需求。经过几轮沟通澄清，实际要的东西是：**一个"合并后
的"覆写脚本文件**——本身就包含了上游最新代码 + 自定义逻辑，下载到 Sub-Store 后
由 Sub-Store 自己生成含真实节点信息的配置，我们的项目从头到尾不接触任何真实
节点数据。

进一步查证发现，上游项目本身就是为这个需求设计的：
- 项目结构是标准 TypeScript 项目（`src/*.ts` 源码 + `esbuild` 构建流水线），
  不是只有一个孤立的 JS 文件
- 官方文档 `docs/HOW_TO_CUSTOMISE.md` 明确写了推荐流程：Fork 仓库 → 直接修改
  `src/*.ts` 源码 → 用项目自带的 GitHub Actions（`npm version patch` 触发）
  自动构建、发布到 `dist` 分支 → 用 `https://cdn.jsdelivr.net/gh/<用户名>/
  override-rules@dist/convert.min.js` 作为最终覆写脚本链接
- 项目还有一份专门写给 AI Agent 的 `AGENTS.md`，说明了核心架构原则（源文件驱动、
  禁止直接改产物文件）和代码规范

## 考虑过的选项 Options Considered
1. **维持 ADR-0001 的方向（独立后处理项目）**
   - 优点：技术上完全自主可控
   - 缺点：从根上误解了需求（用户要的是"一个脚本文件"，不是"一套生成流水线"）；
     且重新发明了上游项目已经提供的能力（TS 源码管理、自动构建发布），纯属
     重复造轮子
2. **Sub-Store 服务端脚本链式调用（`produceArtifact`）**
   - 优点：交付方式贴合现有使用习惯（Sub-Store 生成、Clash Verge 订阅）
   - 缺点：需要把处理逻辑改写成适配 Sub-Store 脚本沙箱的形式（不是完整
     Node.js 环境），且有多个未验证的技术细节（沙箱内网络请求、鉴权等）
3. **Fork 上游仓库，按官方文档在 `src/*.ts` 源码里直接定制（已选定）**
   - 优点：官方支持、有文档、有现成的构建/发布流水线；产出的 `convert.min.js`
     天然包含最新上游代码；不涉及任何真实节点数据，仓库可以公开；后续同步
     上游更新是标准 git fork 操作（`git fetch upstream && git merge`）
   - 缺点：需要熟悉这个项目的源码结构（`constants.ts`/`proxy_groups.ts`/
     `rules.ts`/`rule_providers.ts` 各自负责什么），有一定学习成本，但一次性

## 决定 Decision
采用方案 3。Fork `powerfullz/override-rules`，自定义内容直接写进
`src/*.ts` 源码，用项目自带的构建/发布流程产出最终脚本。

## 理由 Rationale
- 直接命中真实需求："一个包含上游最新代码的合并脚本文件"，不多不少
- 不重复造轮子，复用项目本身已经打磨好的工程能力（TS 源码管理、自动构建、
  自动发布）
- 不涉及真实节点数据，之前纠结的"private 仓库 raw 链接订阅不了""公开仓库
  节点信息泄露"这些问题全部不存在
- 同步上游更新是标准 git 操作，冲突概率低（因为改动的位置正是项目预留给
  定制者的扩展点）

## 与之前决策的关系
- **ADR-0001（链式脚本 vs Fork）**：结论被推翻，改用 Fork。ADR-0001 记录本身
  保留，标记为已废弃，不删除——错误方向本身也是有价值的记录，说明当时基于什么
  信息做的判断、后来因为什么新信息推翻了它
- **ADR-0002（自维护静态资源小名单）**：**决策内容不变，只是实现位置改变**。
  仍然是"移除上游大列表、换成自维护小名单"，只是现在落在
  `src/rule_providers.ts`（定义 `MyDirectCDN` provider）+
  `ruleset/MyDirectCDN.list`（清单内容），而不是独立项目里的 JS 后处理逻辑

## 影响 Consequences
- 好处：架构更简单、更贴合真实需求、维护成本更低（复用上游工程能力）
- 代价：之前 ADR-0001 方向下做的 `custom-override.js`/`build.js`/GitHub Actions
  工作流等，作为独立项目已经没有存在价值，相关经验（问题排查结论、AI 服务
  分组设计等）已经迁移落地到这个 Fork 项目里，原项目可以归档或废弃
- 后续：这个 Fork 项目的记录规则见 `_fork/SOP.md`，不使用仓库根目录的
  `README.md`/`CHANGELOG.md`（那两个文件是上游自己在维护、会随同步更新变动，
  塞自己的内容进去会有合并冲突风险）
